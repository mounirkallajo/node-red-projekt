3. EXAKTE FETCH-REIHENFOLGE IN offlineCachedFetch()
================================================================================

Für MapTiler-kritische Runtime-Ressourcen (Style, tiles.json, Sprite, Glyph):

  Schritt 1: offlineCachedResponseForUrl()  → IndexedDB
  Schritt 2: offlineNativeRuntimeResponseForUrl() → native readCachedResource
  Schritt 3: mapTilerCooldownBeforeRequest()
  Schritt 4a: Wenn cooldown blocked → mapTilerCooldownFallbackResponse() (503)
  Schritt 4b: Wenn nicht blocked → Netzwerk via mapTilerFetchWithTimeout / upstreamFetch
  Schritt 5: Bei Netzwerkfehler oder 401/403/429 → cooldown start → 503

Für nicht-kritische URLs (z.B. Tiles): Schritt 2 wird übersprungen.
Reihenfolge: IndexedDB → Cooldown-Check → Netzwerk → 503.

================================================================================

NEIN — nicht für die gelisteten kritischen Ressourcen.

  -----------------------|--------------------------------------------------

  sprite@2x.json / .png  | gleiche Regex
  font/glyph .pbf        | offlineIsGlyphUrl

Alle sind in offlineIsMapTilerCriticalRuntimeResourceUrl() enthalten.

5. STYLE-FILE-LADEN UND HTTP 503 VOR NATIVE/NETZWERK?
================================================================================

GRÖSSTENTEILS JA — mit einer Präzisierung:

  - Runtime MapLibre Style-Fetch: patched global.fetch → offlineCachedFetch
    → gleiche Reihenfolge. Native wird vor Cooldown/503 versucht.

  - Offline-Region-Download (cacheMapRegion, buildOfflineRegionDownloadPlan):
    lädt Style via global.fetch(styleUrl), NICHT fetchOfflineMapResource.
    Geht aber trotzdem durch offlineCachedFetch (Fetch-Patch) → Fix greift.

  - fetchOfflineMapResource(url) ohne offlineCacheOnly:
    delegiert an offlineCachedFetch → gleiche Reihenfolge, Netzwerk erlaubt.

  - fetchOfflineMapResource(url, { offlineCacheOnly: true }):
    IndexedDB → native → cooldown/503, KEIN Netzwerk (by design für cache-only).

Style-Download mit verfügbarem Netzwerk bekommt kein 503 von Cooldown, solange
Native-Cache den Style hat. Wenn beide Caches fehlen und Cooldown aktiv ist,
ist 503 vor Netzwerk weiterhin möglich — beabsichtigtes Cooldown-Verhalten.


================================================================================
6. readCachedResource() — GLEICHE URL-AUFLÖSUNG WIE hasUrl()?
================================================================================

GRÖSSTENTEILS JA — nicht byte-identisch.

Beide nutzen OfflineMapCacheStore und findCachedUrlByCanonicalBase() für
Alias-Lookup. Storage-Keys über canonicalKey() (entfernt key/apikey/access_token,
sortiert verbleibende Query-Params, normalisiert Glyph-Font-Stacks).

readCachedResource Auflösung (resolveCachedLookupUrl):

  1. store.hasUrl(url) — nutzt canonicalKey(url) für Bin-Pfad
  2. store.hasUrl(stripQueryAndFragment(url)) — Extra-Schritt, den hasUrl() nicht
     explizit macht
  3. findCachedUrlByCanonicalBase(url) → Alias, mit store.hasUrl(alias) und
     alias ≠ url und alias ≠ stripped

  Fall                    | Erwartetes Verhalten
  ------------------------|--------------------------------------------------
  ?key=...                | via canonicalKey / stripped lookup
  %20 in Pfaden           | JS offlineNormalizeGlyphUrl; Java normalizeGlyphCanonicalBase
  Glyph-Komma-Fontstacks  | gleiche Store-Normalisierung
  sprite@2x               | Regex sprite(?:@2x)?\.(json|png)
  tiles.json              | Regex + findCachedUrlByCanonicalBase

REVIEW-HINWEIS (kein bestätigter Bug):
hasUrl() kann true sein, wenn findCachedUrlByCanonicalBase nicht-leer ist, auch
wenn store.hasUrl(url) false ist. readCachedResource kann trotzdem found:false
liefern, wenn alias gleich stripped ist und store.hasUrl(stripped) false ist.
Bei normalen gesunden Cache-Einträgen sollte das nicht vorkommen. Symptom wäre:
MapStyleRestoreDependencyHit source=native bei Preflight, aber
OfflineNativeRuntimeMiss zur Laufzeit.

7. NATIVE METHODE LIEFERT ECHTE BYTES?
================================================================================

JA.

Bei Treffer liefert readCachedResource:

  - found: true
  - mimeType (aus Sidecar oder guessCachedResourceMimeType)
  - base64 (vollständige Datei via Base64.encodeToString(bytes, Base64.NO_WRAP))

Bei Miss: nur found: false (kein base64). Leere/null Bytes → found: false.

Read-only: nutzt readBytes/readContentType; keine Writes, Queue oder Migration.


================================================================================
8. JS DEKODIERT UND LIEFERT ECHTE Response?
================================================================================

JA.

  - base64ToUint8Array() via atob → Uint8Array
  - new Response(nativeRead.bytes, { status: 200, headers: { Content-Type: ... } })
  - Leeres/fehlendes base64 abgelehnt:
    if (!result || !result.found || !result.base64) return { found: false }
  - Ungültiges base64: atob wirft → catch → { found: false }

9. PREFLIGHT MapStyleRestoreDependencyHit source=native → RUNTIME-LOGS?
================================================================================

JA — wenn Runtime durch offlineCachedFetch geht und IndexedDB fehlt.

Erwartete Sequenz für dieselbe URL:

  [offline-map] OfflineNativeRuntimeHit: kind=... url=...
  [offline-map] OfflineRuntimeServedFromNative: ...

Plus bestehende Hit-Logs (OfflineTileJsonHit, OfflineSpriteHit, etc.).

EINSCHRÄNKUNG: Preflight nutzt hasUrl() (Existenz). Runtime nutzt
readCachedResource() (Bytes). Sollten für gesunde Einträge übereinstimmen;
Divergenz nur im Q6-Edge-Case möglich.

10. REQUEST-STORM-RISIKO
================================================================================

Mechanismus                    | Native-Miss gecacht? | Anmerkung
-------------------------------|----------------------|----------------------------------
offlineResourceLog /           | Nur Logs, 10s        | Reduziert Log-Spam, nicht
OfflineNativeRuntimeMiss       | Throttle pro URL     | Bridge-Calls
mapTilerCooldownBlock          | Nur Logs, 5s/URL     | Gleich
offlineKnownRuntimeGlyphMiss   | In-Memory pro        | Überspringt NETZWERK, nicht
                               | kanonischer Glyph-URL| native readCachedResource
Native Negative Cache          | NEIN                 | Jeder Request: IDB miss →
                               |                      | Native-Bridge-Call

RISIKO: MapLibre kann wiederholt dieselbe fehlende Glyph-Range anfragen. Jeder
Request ruft trotzdem native readCachedResource auf. Logs gedrosselt; Bridge-Calls
nicht.

MITIGATION HEUTE: Glyph-Fallback + offlineKnownRuntimeGlyphMiss verhindern
Netzwerk-Stürme; Cooldown begrenzt MapTiler-Netzwerk. KEINE Mitigation für
wiederholte Native-Reads bei persistenten Misses.

Kein funktionaler Bug für das gemeldete 503-Problem; residuelles Performance-
Risiko bei vielen fehlenden Glyphs.
